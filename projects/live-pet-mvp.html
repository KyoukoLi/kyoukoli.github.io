直播虚拟宠物 MVP — 技术方案

MVPv1.02026-03-31

一、MVP 目标

用最小代码跑通核心价值链：一个用户领养宠物 → 喂食升级 → 房间内可见 → 排行榜展示。

设计原则：少即是多。MVP 不追求功能完整，追求核心体验闭环。

二、MVP 功能范围

2.1 包含（To Do）

  • 领养蛋：用户发弹幕「领养一个蛋」→ 系统下发宠物蛋
  • 弹幕加速孵化：房间内任意人发弹幕 → 倒计时减少
  • 宠物出场：孵化完成后，宠物 Lottie 动画出现在直播间挂件位
  • 喂食升级：发弹幕「喂食」→ 宠物饱食度恢复，等级+1
  • 房间排行：显示当前房间内宠物等级 Top 10
  • 主播开播buff：开播时经验加成（+50%），随时间递减

2.2 不包含（Not To Do）

  • ✗ AI 对话能力
  • ✗ 多宠物收藏柜
  • ✗ 稀有度体系
  • ✗ 进化分支
  • ✗ 主播控制台
  • ✗ 跨房间排行
  • ✗ PK 系统

三、核心用户流程

用户A 进直播间
    ↓
发弹幕「领养一个蛋」
    ↓
获得宠物蛋（倒计时 5 分钟）
    ↓
房间内大家发弹幕加速（「加油」「快出来」等）
    ↓
倒计时归零 → 宠物孵化出现
    ↓
用户A 继续发弹幕「喂食」→ 宠物升级
    ↓
宠物等级在房间排行榜上可见
    ↓
主播开播时，房间内所有宠物获得 5 分钟 +50% 经验加成

四、数据模型

4.1 宠物（Pet）

Pet {
  pet_id:        string   // 唯一标识
  user_id:       string   // 所属用户
  room_id:       string   // 当前所在房间（停播时空）
  name:          string   // 宠物名（默认随机）
  level:         int      // 当前等级（初始 1）
  exp:           int      // 当前经验值
  exp_to_next:   int      // 升级所需经验（固定阈值）
  hunger:        float    // 饱食度 0.0~1.0
  state:         enum     // egg | hatching | active | sleeping
  hatched_at:    int      // 孵化时间戳
  created_at:    int      // 创建时间戳
}

4.2 房间（Room）

Room {
  room_id:         string   // 房间 ID
  host_id:         string   // 主播 ID
  is_live:         bool     // 是否正在直播
  live_started_at: int      // 开播时间戳
  egg_hatch_speed: int      // 全房间孵化加速计数
}

4.3 经验值计算

操作经验值备注
发弹幕「喂食」+10 XP基础行为
收到礼物（任意）+礼物价值 × 10 XP按快手礼物价值换算
主播开播中以上全部 × 1.5开播 buff（5 分钟内）
开播 5-30 分钟以上全部 × 1.2buff 衰减
开播 30 分钟以上以上全部 × 1.1buff 进一步衰减

升级所需经验：level × 100 XP（Lv.1→Lv.2 需要 100 XP，Lv.2→Lv.3 需要 200 XP，以此类推）

4.4 孵化加速计算

每条弹幕减少倒计时 3 秒
上限：最多 60 条弹幕（即最多加速 180 秒 = 3 分钟）
原始倒计时：5 分钟（300 秒）
最快孵化：2 分钟（300 - 180）

5.1 弹幕命令解析

func parseCommand(text string) Command {
  switch text {
  case "领养一个蛋":         return CMD_ADOPT
  case "喂食":               return CMD_FEED
  default:                   return CMD_ACCELERATE  // 其余所有弹幕均加速孵化
  }
}

5.2 关键实现细节

  • 弹幕防刷：同一用户每 3 秒最多触发 1 次，避免刷屏
  • 宠物休眠：饱食度归零后宠物进入「sleeping」状态，不显示动画，只在喂食后恢复
  • 房间状态：主播停播时，is_live=false，buff 消失，宠物状态冻结（不消失）
  • 排行榜：每次喂食/收礼物后更新 Redis SortedSet，渲染层每 5 秒拉取一次
  • Redis 键设计：
    • pet:{pet_id} — Hash，宠物全量数据
    • room:{room_id}:rank — SortedSet，分数=level，用户ID=member
    • room:{room_id}:info — Hash，房间状态

六、部署方案（推荐）

组件推荐方案说明
宠物引擎Railway / Zeabur 单节点轻量 Go 服务，高并发弹幕需 Kafka
状态存储Redis Cloud减少自运维，支持 1M QPS
动画渲染快手小窗 / AR 挂件需快手开放平台审核接入
LLM（可选）MiniMax / 智谱 GLMMVP 阶段不需要，后续加

七、MVP 里程碑

阶段目标验收标准
M1本地跑通领养→孵化→喂食→升级闭环本地脚本模拟弹幕事件，宠物从 Lv.1 到 Lv.3
M2接入快手测试环境 WebSocket真实弹幕能触发宠物行为
M3渲染层接入（模拟挂件）宠物动画和排行在页面可见
M4灰度 1 个真实主播直播 30 分钟，验证互动数据

八、风险与对策

风险概率对策
快手开放平台不支持弹幕 WebSocket 订阅中改用轮询 API 或申请白名单权限
直播间挂件接入门槛高（需审核）中MVP 阶段先用 H5 页面模拟，嵌入 OBS 推流
弹幕防刷失效导致经验值被刷低Redis 计数 + 滑动窗口限速
Redis 宕机宠物数据丢失低宠物数据持久化到 MySQL（每小时同步）
